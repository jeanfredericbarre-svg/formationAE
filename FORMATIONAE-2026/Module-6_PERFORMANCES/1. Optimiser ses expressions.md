## Pourquoi les expressions peuvent ralentir AE 

Rappel du Module 1 : une expression est **réévaluée à chaque frame**. Sur une composition de 30 secondes à 25fps, c'est 750 évaluations par expression. Multiplie par le nombre de calques et de propriétés concernées, et certaines compositions peuvent devenir très lentes à prévisualiser.

Les expressions les plus coûteuses sont celles qui :
- font des **lectures dans d'autres calques** (`thisComp.layer(...)`)
- utilisent `sourceRectAtTime()` très gourmande
- contiennent des boucles ou des calculs complexes
- s'appliquent à un **grand nombre de calques** (texte avec de nombreux caractères, Repeater avec beaucoup de copies)

## posterizeTime() : réduire la fréquence d'évaluation 

`posterizeTime(fps)` est une fonction AE qui force l'expression à être évaluée à une **fréquence réduite**, indépendamment du framerate de la composition.

```javascript
posterizeTime(10);
wiggle(3, 40)
// Le wiggle est recalculé seulement 10 fois par seconde
// au lieu de 25fps (ou 30fps selon la comp)
// → performance améliorée, au prix d'une légère saccade visuelle
```

**C'est une décision artistique autant que technique.** À 10fps, le wiggle a un aspect plus "stop motion" ou "sketch animé". À 5fps, l'effet est encore plus marqué. C'est parfois exactement l'effet recherché.

```javascript
posterizeTime(12);
// 12fps = un aspect "animation traditionnelle" bien connu
```

> 💡 `posterizeTime()` doit être placé **au début de l'expression**, avant les calculs qu'on veut affecter. Il affecte toute l'expression jusqu'à la fin.

**Autre usage : synchroniser plusieurs expressions :**

```javascript
posterizeTime(2);
seedRandom(index, true);
wiggle(2, 30)
// Réévalué seulement 2 fois par seconde — toutes les expressions
// avec le même posterizeTime(2) changent exactement au même moment
// Effet visuel "quantifié" cohérent sur tous les calques
```

## Identifier les expressions lentes 

**La méthode de diagnostic :**

1. Désactiver toutes les expressions d'un coup : dans la timeline, `F8` pour activer/désactiver le mode expression global, si la performance s'améliore drastiquement, les expressions sont en cause
2. Réactiver calque par calque pour isoler le coupable
3. Sur le calque problématique, désactiver les expressions propriété par propriété (clic sur l'icône `=` dans la timeline)

**Signaux d'alerte courants :**

```
sourceRectAtTime()    → très coûteux, à utiliser avec parcimonie
comp().layer()        → traverser les compositions est lent
Boucles for/while     → évaluation lourde à chaque frame
thisComp.numLayers    → compter les calques à chaque frame
```

<span id="bonnespratiques"></span>
## Bonnes pratiques d'écriture 

**1. Stocker les résultats dans des variables**

Ne calcule pas la même chose plusieurs fois dans la même expression :

```javascript
// ❌ Inefficace — thisComp.layer("CONTROLS") est résolu deux fois :
const v1 = thisComp.layer("CONTROLS").effect("Slider 1")("Curseur");
const v2 = thisComp.layer("CONTROLS").effect("Slider 2")("Curseur");

// ✅ Plus efficace — le calque est résolu une seule fois :
const ctrl = thisComp.layer("CONTROLS");
const v1 = ctrl.effect("Slider 1")("Curseur");
const v2 = ctrl.effect("Slider 2")("Curseur");
```

**2. Éviter les calculs inutiles hors de la plage active**

```javascript
// Si le calque n'est pas encore apparu, retourner la valeur par défaut :
if (time < inPoint) {
  value   // valeur d'origine, pas de calcul
} else {
  wiggle(3, 40)
}
```

**3. Utiliser `value` plutôt que de recalculer**

`value` contient la valeur courante de la propriété sans expression, c'est une lecture directe, bien plus rapide qu'une nouvelle résolution de chemin.

```javascript
// ✅ value[0] est instantané
[value[0], wiggle(3, 40)[1]]

// ❌ Lire thisLayer.transform.position[0] est plus lent
[thisLayer.transform.position[0], wiggle(3, 40)[1]]
```

**4. `posterizeTime()` en dernier recours**

Si une expression est lente et que son rendu n'a pas besoin d'être fluide à 25fps, `posterizeTime()` est le levier le plus simple. À utiliser après avoir optimisé le code lui-même.
