## Deux référentiels, une source de confusion 

Dans AE, toute valeur de position ou de transformation existe dans un **référentiel**, un système de coordonnées qui définit "par rapport à quoi" la valeur est mesurée.

Il en existe deux :

- **Local space** : les coordonnées sont relatives au calque parent
- **World space** : les coordonnées sont relatives à la composition entière

> 💡 Analogie : si tu es dans un train et que tu bouges de 3 pas vers l'avant, tu t'es déplacé de 3 pas en **local space** (par rapport au train). Mais par rapport au sol (world space), tu t'es déplacé de 3 pas + la vitesse du train. Les deux mesures sont vraies, elles parlent simplement de référentiels différents.

Quand il n'y a pas de parenté, local space et world space sont identiques. C'est dès qu'on introduit un parent que la distinction devient critique.

---
<span id="localspace"></span>
## Le local space 

Les valeurs affichées dans la timeline AE sont **toujours en local space**. La propriété Position d'un calque indique sa position *par rapport à son parent*, et si le calque n'a pas de parent, par rapport à la composition.

```
Null parent → Position [500, 300]
  Calque enfant → Position [0, 0]
```

Dans cet exemple, le calque enfant affiche `[0, 0]` dans la timeline, mais il est visuellement positionné à `[500, 300]` dans la composition, parce qu'il hérite de la position de son parent.

Si tu lis `thisLayer.transform.position` dans une expression sur le calque enfant, tu obtiens `[0, 0]` la valeur locale, pas la position réelle dans la composition.

---
<span id="worldspace"></span>
## Le world space 

Le world space est la position **réelle dans la composition**, tous parents confondus. C'est ce que tu vois visuellement à l'écran.

Pour obtenir la position en world space depuis une expression, AE fournit la méthode `toComp()` :

```javascript
thisLayer.toComp([0, 0])
// → retourne la position du point [0,0] du calque (son pivot)
//   exprimée dans les coordonnées de la composition
```

`toComp()` prend un point en **local space** et retourne son équivalent en **world space**.

```javascript
// Position réelle dans la comp du pivot d'un calque :
thisLayer.toComp([0, 0])

// Position réelle du centre d'un calque (en tenant compte de l'ancre) :
thisLayer.toComp(thisLayer.transform.anchorPoint)
```

<span id="probleme"></span>
## Pourquoi ça pose problème avec les expressions 

Le cas le plus fréquent : tu veux qu'un calque B suive la position d'un calque A, mais ils ont des parents différents.

**Le problème :**
```javascript
// Sur la propriété Position du calque B :
thisComp.layer("Calque A").transform.position
// → retourne la position locale de A — pas sa position dans la comp
// Si A a un parent, ce résultat est faux pour B
```

**La solution :**
```javascript
// On lit la position de A en world space...
const posA = thisComp.layer("Calque A").toComp([0, 0]);
// ...puis on la convertit en local space pour B
thisLayer.fromComp(posA)
```

`fromComp()` fait l'inverse de `toComp()` elle prend un point en world space et le retourne en local space du calque courant.

```javascript
thisLayer.fromComp([x, y])
// → "quelle valeur locale dois-je donner à ma propriété Position
//    pour que mon pivot se retrouve à [x, y] dans la composition ?"
```

> 💡 Retiens les deux méthodes ensemble : `toComp()` sort du local vers le world, `fromComp()` rentre du world vers le local. Dans la pratique, on les utilise presque toujours en tandem — lire en world space depuis la source, convertir en local space pour la cible.
